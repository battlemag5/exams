## Лекция 1: История, классы и инкапсуляция
### История
* **1972:** Создание языка C.
* **1980:** "C с классами" (Бьерн Страуструп в Bell Labs), добавление возможностей Simula в C.
* **Стандарты:** C++98, C++03, C++11 (C++0x), C++14, C++17, C++20, C++23, C++26.

### Переход от C к C++
* **Проблемы C:** Сложно создавать абстракции, бедная стандартная библиотека, отсутствие связи между данными (struct) и кодом, который с ними работает.
* **Решение C++:** Перенос функций внутрь структуры. Структура теперь содержит не только поля, но и логику.

### Ключевые понятия
* **Указатель `this`:** Невидимый первый аргумент любого метода класса, указывающий на текущий экземпляр объекта. Это "zero-cost abstraction" (абстракция с нулевой стоимостью).
* **Инкапсуляция:**
    * Объединение данных (полей) и логики (методов).
    * Скрытие внутренних деталей реализации для поддержания **инвариантов** (условий, которые всегда должны быть истинными для корректной работы объекта).
* **Модификаторы доступа:и**
    * `private`: доступ только из кода самого класса. Используется для защиты инвариантов.
    * `public`: публичный API, доступен отовсюду.
* **struct vs class:**
    * В `struct` доступ по умолчанию `public`.
    * В `class` доступ по умолчанию `private`.
    * Рекомендуется использовать `struct` для простых наборов данных без инвариантов.

### Жизненный цикл объекта
* **Конструктор (`Constructor`):** Вызывается при создании объекта для инициализации полей. Может быть несколько конструкторов с разными аргументами.
* **Деструктор (`Destructor`):** Вызывается автоматически при уничтожении объекта. Используется для освобождения ресурсов (например, памяти). Имя: `~ClassName()`.
* **Оператор `new`:** Выделяет память в куче, проверяет результат (кидает исключение при ошибке), вызывает конструктор и возвращает указатель.
* **Оператор `delete`:** Вызывает деструктор и освобождает память.
* **`delete[]`:** Используется специально для массивов, выделенных через `new[]`, вызывает деструкторы для каждого элемента.

### Время жизни объектов (Storage Duration)
1. **static:** до завершения программы (выход из `main` или `exit`).
2. **automatic:** до конца области видимости (scope).
3. **dynamic:** до явного вызова `delete`.

## Лекция 2: Указатели, ссылки и const
### Указатели (Pointers)
* **Определение:** Переменная, хранящая адрес другой переменной.
* **Операции:**
    * `&a`: взятие адреса.
    * `*pa`: разыменование (доступ к значению по адресу).
* **Арифметика указателей:** При сложении указателя с числом прибавляется `n * sizeof(T)`.
* **Безопасность:** Указатели небезопасны (возможны двойное освобождение, доступ по неверному адресу).
* **nullptr:** (C++11) Специальное значение для "нулевого" указателя. Имеет тип `nullptr_t`. Лучше, чем `0` или `NULL`.

### Ссылки (lvalue references)
* **Определение:** Альтернативное имя (псевдоним) для существующей переменной.
* **Особенности:**
    * Должны быть инициализированы при создании.
    * Не могут быть перенаправлены на другой объект.
    * Не являются самостоятельными объектами (нет адреса ссылки, нет массивов ссылок).
    * Безопаснее указателей (меньше шаблонного кода, сложнее ошибиться).
* **Реализация:** Внутри обычно реализуются как скрытые указатели.

### Ключевое слово const
* **Значение:** Ограничение на изменение и одновременно разрешение (например, для временных объектов).
* `const int* p`: указатель на константу (данные менять нельзя, указатель можно).
* `int* const p`: константный указатель (данные менять можно, указатель нельзя).
* `const int* const p`: константный указатель на константу.
* **Константные ссылки:** Могут привязываться к временным объектам.

### Временные объекты и время жизни
* **Временные объекты:** Создаются компилятором (например, результат `a + b`). Живут до конца полного выражения (обычно до `;`).
* **Продление жизни (Lifetime extension):** Если временный объект привязан к константной `lvalue` ссылке, его жизнь продлевается до времени жизни этой ссылки.
* **"Висячие" (Dangling) объекты:** Указатели или ссылки на уже уничтоженные объекты. Классическая ошибка — возврат ссылки на локальную переменную.

## Лекция 3: Инициализация, копирование и Rule of 3
### Виды инициализации
1. **Default initialization:** Для примитивов — мусор (UB). Для классов — вызов конструктора по умолчанию.
2. **Value initialization (`T()`):** Для примитивов — обнуление. Для классов — конструктор по умолчанию.
3. **Direct initialization (`T v(args)` или `T v{args}`):** Прямой вызов конструктора.
4. **Copy initialization (`T v = other`):** Создание копии.
5. **Aggregate initialization (`T v = {val1, val2}`):** Для "агрегатов" (массивы, структуры без private полей, пользовательских конструкторов и наследования).
6. **List initialization (`{}`):** Помогает избежать "the most vexing parse" (ошибочной интерпретации объявления переменной как функции).

### Тонкости конструкторов
* **Список инициализации (Member initializer list):** `Constructor(): field(val) {}`.
    * Рекомендуется вместо присваивания в теле.
    * Поля инициализируются в порядке их **объявления в классе**, а не в списке.
    * Обязателен для `const` полей и ссылок.
* **Делегирующие конструкторы (C++11):** Один конструктор может вызывать другой: `Ctor1(): Ctor2(args) {}`.
* **Ключевое слово explicit:** Запрещает неявное преобразование типов через конструктор с одним аргументом.

### Копирование
* **Конструктор копирования (Copy Constructor):** `T(const T& other)`. Вызывается при создании копии (передача в функцию по значению, возврат из функции).
* **Оператор копирующего присваивания (Copy Assignment Operator):** `T& operator=(const T& other)`. Вызывается при присваивании одного существующего объекта другому. Должен обрабатывать самоприсваивание.
* **Правило трех (Rule of Three):** Если вам нужно реализовать один из трех методов (деструктор, конструктор копирования или оператор присваивания), скорее всего, вам нужно реализовать все три.
* **NRVO (Named Return Value Optimization):** Оптимизация, позволяющая избежать лишнего копирования при возврате локального объекта из функции.

## Лекция 4: Операторы, друзья, идиома RAII, Rule of 5

### Операторы
* **Перегрузка операторов:** Позволяет определить поведение операторов (`+`, `-`, `*`, `=`, `<<`, `[]`, `()`, и т.д.) для пользовательских типов.
* `operator+`, `operator-`, `operator*`, `operator/`: бинарные арифметические операторы.
* `operator+=`, `operator-=`: составные операторы присваивания.
* `operator<<`, `operator>>`: операторы побитового сдвига или потокового ввода/вывода.
* `operator[]`: доступ по индексу (для контейнеров).
* `operator()`: функтор (function object). Позволяет объекту вести себя как функция.
* `operator T()`: оператор преобразования типа (конверсии).
* `operator++`, `operator--`: префиксный (`++x`) и постфиксный (`x++`) инкремент/декремент. Постфиксный отличается фиктивным аргументом `int`: `T operator++(int)`.
* **Дружественные функции (friend):** Функция или класс, не являющийся членом данного класса, но имеющий доступ к его `private` полям. Объявляется с ключевым словом `friend`.
* **Выбор: метод vs свободная функция:** Если оператор модифицирует объект (например, `+=`) — делается методом. Если создает новый объект из двух (например, `+`) — обычно свободной функцией (чтобы работала симметрия типов, например, `int + MyClass`). Операторы `<<` и `>>` для потоков делаются свободными функциями (первый аргумент — поток, не наш тип).

### Идиома RAII (Resource Acquisition Is Initialization)
* **Определение:** Ключевая идиома C++: захват ресурса производится в конструкторе, освобождение — в деструкторе.
* **Гарантия:** Ресурс будет корректно освобожден при выходе из области видимости (даже при исключении — благодаря раскрутке стека).
* **Примеры:** `std::string` (управление памятью), `std::vector` (динамический массив), `std::ifstream` (файловый дескриптор), умные указатели.
* **Ресурсы:** Память, файловые дескрипторы, сокеты, мьютексы, соединения с БД и т.д.

### Правило пяти (Rule of Five)
* C++11 добавил move-семантику, расширив правило трех до правила пяти:
  1. Деструктор (`~T()`)
  2. Конструктор копирования (`T(const T&)`)
  3. Оператор копирующего присваивания (`T& operator=(const T&)`)
  4. **Конструктор перемещения (Move Constructor):** `T(T&&)`
  5. **Оператор перемещающего присваивания (Move Assignment):** `T& operator=(T&&)`

### Конструктор перемещения
* **Семантика:** "Крадет" ресурсы у временного объекта (rvalue), а не копирует их.
* **Параметр:** rvalue-ссылка (`T&&`).
* **Состояние после перемещения:** Объект-источник должен остаться в корректном (но не определенном) состоянии — обычно обнуляется, чтобы его деструктор не освободил украденные ресурсы.
* **Когда вызывается:** При возврате из функции, при передаче временного объекта.

### std::move
* **Определение:** `std::move` — это просто приведение типа к rvalue-ссылке (`static_cast<T&&>`). Ничего не перемещает.
* **Назначение:** Позволяет "разрешить" перемещение, говоря компилятору: "этот объект можно рассматривать как временный".

### Копирование vs Перемещение
* **std::string:** Копирование O(n), перемещение O(1) (копирование указателя).
* Рекомендуется всегда проверять: действительно ли нужно копирование, или можно переместить?

## Лекция 5: Массивы, своп, идиомы копирования/перемещения

### Массивы в C и C++
* **Массивы в C:** Встроенные массивы (`int arr[10]`) — это просто contiguous block of memory (непрерывный блок памяти). Знают свой размер на этапе компиляции (`
* **Массивы в C++:** `std::array` (фиксированный размер, известен на этапе компиляции, знает свой `.size()`) и `std::vector` (динамический размер, растет при необходимости).
* **Проблема `sizeof`:** Для встроенных массивов `sizeof` работает только в той же области видимости. При передаче в функцию массив "распадается" в указатель, и `sizeof` вернет размер указателя.
* **Массивы и инициализация:** Массивы можно инициализировать через aggregate initialization: `int a[] = {1, 2, 3};` или `int a[5]{1, 2, 3};` (остальное обнулится).
* **Массивы в классах:** В классе можно хранить массив фиксированного размера, если размер известен на этапе компиляции. В противном случае использовать `std::vector`.

### Идиома swap
* **Зачем:** Для реализации move-семантики и copy-and-swap идиомы. Обмен двух объектов должен быть быстрым и не кидать исключений.
* **Реализация:** Обычно через `std::swap` (поэлементный обмен) или через пользовательский метод `Swap()`.
* **noexcept:** Операция swap обязательно помечается `noexcept`. Это важно: если swap может кинуть исключение, то copy-and-swap идиома не будет безопасной.
* **std::swap:** По умолчанию создает временную копию. Для пользовательских типов лучше перегрузить или использовать move-семантику:
```cpp
void swap(T& a, T& b) noexcept {
    T tmp(std::move(a));
    a = std::move(b);
    b = std::move(tmp);
}
```

### Copy-and-Swap идиома
* **Идея:** Оператор присваивания реализуется через конструктор копирования + swap. Принимаем параметр **по значению** (сработает копирование или перемещение):
```cpp
T& operator=(T other) noexcept {
    swap(*this, other);
    return *this;
}
```
* **Преимущества:**
  * Обрабатывает самоприсваивание автоматически (делаем копию, потом меняемся).
  * Одна реализация работает и для копирующего, и для перемещающего присваивания.
  * Если конструктор копирования кинул исключение, исходный объект не изменился (strong exception guarantee).
* **Недостатки:** Может быть менее эффективным, чем раздельные операторы (лишнее копирование, если объект уже готов к приему перемещением).

### Правило нуля (Rule of Zero)
* **Идея:** Если ваш класс не управляет ресурсами напрямую, **не пишите** деструктор, конструкторы копирования/перемещения и присваивания. Компилятор сгенерирует их автоматически.
* **Как достичь:** Используйте RAII-обертки для ресурсов: `std::vector`, `std::string`, `std::unique_ptr`, `std::shared_ptr`.
* **Смысл:** Чем меньше ручное управление памятью, тем меньше ошибок. Правило нуля — это цель, к которой стоит стремиться.
* **Правило пяти применяется, только если нужно явное управление ресурсами.**

### Виртуальные деструкторы
* **Проблема:** При удалении объекта производного класса через указатель на базовый класс без виртуального деструктора — поведение не определено (UB).
* **Решение:** Если класс предназначен для наследования (имеет хотя бы один виртуальный метод), деструктор должен быть виртуальным: `virtual ~Base() = default;`
* **Исключение:** Если базовый класс является интерфейсом (только чисто виртуальные методы), деструктор может быть чисто виртуальным, но с телом: `virtual ~IBase() = 0;` (тело нужно, так как деструкторы вызываются в обратном порядке и дойдут до базового).
* **Практика:** Если вы пишете класс, не предназначенный для наследования, можно запретить наследование (`final`) и не делать деструктор виртуальным (меньше оверхед — не нужна vtable-запись).

### Ключевые слова = default и = delete
* **= default:** Явно указать компилятору сгенерировать стандартную реализацию (конструктор, деструктор и т.д.). Полезно, если мы объявили другой конструктор (по умолчанию не сгенерируется), но хотим оставить стандартное поведение для какого-то метода.
* **= delete:** Запретить использование определенного метода. Например, запретить копирование: `T(const T&) = delete;`. Или запретить определенные перегрузки: `void foo(int) = delete;`.

### Спецификатор const для методов
* **const-методы:** Методы, которые не изменяют состояние объекта. Объявляются с `const` после списка параметров: `void method() const;`
* **Перегрузка по const:** Можно иметь две версии метода — const и не-const. Для константных объектов вызовется const-версия, для неконстантных — не-const.
* **Возврат из const-методов:** Если метод возвращает ссылку на внутренние данные, const-версия должна вернуть `const T&`, а не-const версия — `T&`.
* **Важно:** const-метод **не может** изменять поля класса (кроме полей, помеченных `mutable`). Это контракт для пользователя класса.
* **mutable:** Позволяет изменять поле даже в const-методе. Используется для кэширования, мьютексов, счетчиков вызовов и т.д.

### lvalue и rvalue
* **lvalue (locator value):** Объект, у которого есть адрес в памяти. Может стоять слева от присваивания. Примеры: переменные, элементы массива, разыменованный указатель.
* **rvalue (read value):** Временный объект, не имеющий постоянного адреса. Не может стоять слева от присваивания. Примеры: литералы (`42`), результат выражения (`a + b`), временный объект (`T()`).
* **Ссылки:**
  * **lvalue reference (`T&`):** Ссылка на lvalue. Не может привязаться к rvalue.
  * **const lvalue reference (`const T&`):** Может привязаться к чему угодно (lvalue, rvalue, временный объект). Продлевает жизнь временного объекта.
  * **rvalue reference (`T&&`):** Ссылка на rvalue. Может привязаться только к rvalue. Используется для move-семантики и perfect forwarding.
* **Срок жизни временных объектов (rvalue):** Живут до конца полного выражения (`;`), если не привязаны к `const T&` или `T&&` (продление жизни).

### std::swap и move-семантика
* **Реализация через move:** Современный `std::swap` для пользовательских типов оптимален, если есть move-конструктор и move-присваивание. Три перемещения — дешево.
* **noexcept:** Если ваш move-конструктор и move-присваивание — `noexcept`, то `std::swap` будет использовать move. Иначе — копирование (из-за гарантий безопасности исключений в STL).

## Лекция 6: Наследование, виртуальные методы, абстрактные классы

### Наследование
* **Синтаксис:** `class Derived : public Base { ... };`
* **Отношение "is-a":** Производный класс ЯВЛЯЕТСЯ разновидностью базового. Применимо, когда Derived можно использовать везде, где ожидается Base.
* **Видимость при наследовании:**
  * `public` наследование: public-поля Base остаются public в Derived. Самый распространенный вид.
  * `protected` наследование: public и protected поля Base становятся protected в Derived.
  * `private` наследование: public и protected поля Base становятся private в Derived (частный случай — композиция через наследование).
* **Порядок конструирования:** Сначала вызывается конструктор базового класса, потом конструкторы полей, потом тело конструктора производного класса. Деструктор — в обратном порядке.
* **Вызов конструктора базы:** Если у Base нет конструктора по умолчанию, нужно явно вызвать его в списке инициализации Derived: `Derived(params) : Base(params), field(val) {}`
* **Срезка (Slicing):** При передаче по значению (например, `void foo(Base obj)`) производный класс "срезается" до базовой части. Теряются поля и методы производного класса. Исправляется передачей по ссылке или указателю.

### Виртуальные методы
* **Проблема:** При вызове метода через указатель/ссылку на Base может вызваться метод Base, даже если реальный тип объекта — Derived.
* **Решение:** Виртуальные методы. Если метод объявлен как `virtual` в Base, то при вызове будет вызван метод Derived (если он переопределен).
* **override (C++11):** Явно указывает, что метод переопределяет виртуальный метод базового класса. Ошибка компиляции, если в базе нет такого виртуального метода.
* **final (C++11):** Запрещает дальнейшее переопределение виртуального метода или наследование от класса:
  * `void method() override final;` — запретить переопределение в подклассах.
  * `class Derived final : public Base { ... };` — запретить наследование от Derived.
* **Виртуальный деструктор:** Если класс предназначен для наследования, деструктор должен быть виртуальным.
* **Чисто виртуальные методы и абстрактные классы:** `virtual void method() = 0;`. Класс, имеющий хотя бы один чисто виртуальный метод, является абстрактным — нельзя создать его экземпляр. Производные классы должны реализовать все чисто виртуальные методы, чтобы быть конкретными.

### Таблица виртуальных методов (vtable)
* **Определение:** Для каждого класса с виртуальными методами (или унаследовавшего их) компилятор создает статическую таблицу — vtable. В ней хранятся указатели на виртуальные функции.
* **Указатель vptr:** Каждый объект такого класса содержит скрытый указатель (vptr) на vtable своего класса (обычно в начале объекта, до полей).
* **Вызов виртуального метода:** Фактически: `obj->vptr->method_table[index]()`. Два косвенных обращения.
* **Размер объекта:** Увеличивается на размер указателя (vptr). На 64-битных системах — 8 байт.
* **Несколько базовых классов:** При множественном наследовании объект может содержать несколько vptr (по одному на каждую базу с виртуальными методами).
* **Стоимость:** vtable lookup — небольшой накладной расход (по сравнению с обычным вызовом). Это не zero-cost abstraction, но обычно overhead пренебрежимо мал. Виртуальные вызовы не могут быть встроены (inlined).

### Итоги по наследованию
* **Композиция предпочтительнее наследования:** "Favor composition over inheritance". Наследование часто ведет к хрупкому коду и сильной связности.
* **Наследование vs интерфейсы:** В C++ нет отдельной конструкции "интерфейс". Интерфейс моделируется абстрактным классом со всеми чисто виртуальными методами.
* **protected:** Промежуточный доступ: поля видны в классе и в производных классах, но не снаружи. Нарушает инкапсуляцию, так как создает зависимость производных классов от деталей реализации базы.

## Лекция 7: Исключения

### Стек и куча (Stack vs Heap)
* **Стек (Stack):** Каждый поток имеет свой стек (обычно 1-8 МБ). Локальные переменные хранятся на стеке. Выделение/освобождение — очень быстро (просто сдвиг указателя стека). Память автоматически освобождается при выходе из области видимости.
* **Куча (Heap):** Глобальная память, доступная всем потокам. Выделение (`new`/`malloc`) и освобождение (`delete`/`free`) значительно медленнее. Требует ручного управления (или RAII-оберток). Память фрагментируется.

### Обработка ошибок в C
* **errno:** Глобальная переменная, устанавливаемая при ошибке. Проблемы: не thread-safe (в современных системах — TLS), легко забыть проверить, легко перезатереть.
* **Проверка кода возврата:** Функции возвращают специальное значение (например, `nullptr`, `-1`). Проблемы: загромождает код, можно забыть проверить.
* **setjmp/longjmp:** Позволяют прыгать между функциями. Проблемы: не вызывают деструкторы (пропускают раскрутку стека), опасны в C++.

### Исключения в C++
* **Синтаксис:**
  * `try { /* код, который может кинуть исключение */ }`
  * `catch (Type e) { /* обработка исключения */ }`
  * `throw SomeException();` — выброс исключения.
* **По значению:** Исключение копируется в специальную область памяти ("исключительная память", reserved), поэтому даже если в куче нет места, исключение может быть выброшено (если зарезервировано достаточно места).
* **По ссылке:** Ловить исключения рекомендуется по константной ссылке: `catch (const std::exception& e)`. Это избегает срезки (slicing) при полиморфных исключениях.

### Раскрутка стека (Stack Unwinding)
* **Определение:** При выбрасывании исключения компилятор автоматически уничтожает все локальные объекты (вызывает деструкторы) по пути от `throw` до подходящего `catch`.
* **Порядок:** Деструкторы вызываются в обратном порядке создания объектов.
* **Важно:** Исключение, покинувшее деструктор во время раскрутки стека, вызывает `std::terminate` (аварийное завершение программы). Деструкторы должны быть `noexcept`.

### Гарантии безопасности исключений (Exception Safety Guarantees)
1. **No guarantee:** Нет никаких гарантий. При исключении возможна утечка ресурсов или некорректное состояние.
2. **Basic guarantee:** При исключении объект остается в корректном, но не определенном состоянии. Инварианты соблюдены, утечек ресурсов нет.
3. **Strong guarantee:** При исключении состояние программы не меняется (атомарность — commit/rollback). Достигается через copy-and-swap идиому.
4. **Nothrow guarantee:** Операция никогда не кидает исключения. Используется для деструкторов, swap, move-операций (если помечено `noexcept`).

### noexcept
* **Спецификатор:** `void func() noexcept;` — обещание, что функция не кинет исключение.
* **Оператор:** `noexcept(expression)` — проверка во время компиляции, может ли выражение кинуть исключение. Возвращает `true`/`false`.
* **Оптимизация:** Компилятор может генерировать более эффективный код для `noexcept` функций (не нужен механизм раскрутки стека).
* **Условный noexcept:** `void func() noexcept(condition);`. Часто используется: `noexcept(std::is_nothrow_swappable_v<T>)` для `swap`.
* **Move-конструкторы и swap:** Должны быть `noexcept`. Иначе `std::vector` будет копировать, а не перемещать элементы при реаллокации (из-за гарантий безопасности).

### Иерархия стандартных исключений
* `std::exception` — базовый класс. Виртуальный метод `what()` возвращает `const char*` с описанием ошибки.
* `std::logic_error` — ошибки, которые можно было избежать на этапе написания кода (наследуется от `exception`):
  * `std::invalid_argument` — неверный аргумент функции.
  * `std::domain_error` — значение вне допустимой области.
  * `std::length_error` — превышение допустимого размера (например, `resize` строки).
  * `std::out_of_range` — выход за границы (например, `at()` у вектора).
* `std::runtime_error` — ошибки, которые возникают во время выполнения (наследуется от `exception`):
  * `std::range_error` — результат вычисления вне допустимого диапазона.
  * `std::overflow_error` — арифметическое переполнение.
  * `std::underflow_error` — арифметическое исчерпание (underflow).
* `std::bad_alloc` — не удалось выделить память (кидается `new`). Можно обработать, чтобы не падать при OOM.
* `std::bad_cast` — некорректный `dynamic_cast` на ссылку.
* `std::bad_weak_ptr` — создание `shared_ptr` из `weak_ptr`, который уже истек.
* `std::bad_function_call` — вызов пустого `std::function`.

### Когда использовать исключения vs коды ошибок
* **Исключения:** Когда ошибка не может быть обработана на месте и должна быть передана выше по стеку. Когда ошибка редкая (не влияет на производительность в нормальном потоке). При перегрузке операторов (они не могут вернуть код ошибки).
* **Коды ошибок (expected/std::optional):** Когда ошибка — часть нормального потока работы (например, "файл не найден" — ожидаемое поведение). Когда производительность критична в цикле. Когда важна строгая гарантия времени (embedded, real-time системы).
* **std::expected (C++23):** Альтернатива исключениям, возвращающая либо значение, либо ошибку как часть сигнатуры функции.

### Стоимость исключений
* **Zero-cost when not thrown:** При нулевом выбросе (normal execution) исключения не имеют накладных расходов на производительность (в современных реализациях). Нет проверок на каждом шаге.
* **Размер бинарника:** Увеличивается (таблицы раскрутки стека, Landing Pad'ы — точки входа для обработки исключений в каждой функции).
* **При выбрасывании:** Дорого (walking стека, поиск catch-обработчика, вызов деструкторов). Исключения — для редких ошибок, не для контроля потока.
* **Механизм "Table-driven":** Современные компиляторы используют таблицы (например, Itanium C++ ABI), а не проверки на каждом этапе. Информация о раскрутке хранится отдельно от кода.

* **-fno-exceptions:** Флаг компиляции, отключающий поддержку исключений. Используется во встраиваемых системах и некоторых игровых движках. Стандартная библиотека все еще может кидать исключения.

## Лекция 8: Шаблоны (Templates)

### Введение в шаблоны
* **Определение:** Шаблоны (templates) — механизм C++ для написания обобщенного кода, работающего с разными типами. Позволяют определять функции и классы, параметризованные типом (или значением).
* **Параметры шаблона:** Указываются в угловых скобках: `template <typename T>` или `template <class T>`.
* **typename vs class:** Абсолютно эквивалентны. `typename` появился позже, чтобы избежать путаницы с class. В современных проектах `typename` предпочтительнее для параметров-типов.
* **Компиляция:** Шаблон — не код, а инструкция для компилятора по генерации кода. Код генерируется (инстанцируется) только при подстановке конкретных типов. Отсюда требования: определение шаблона обычно должно быть в заголовочном файле (или visible на месте использования).

### Function Templates (Шаблоны функций)
* **Синтаксис:**
  ```cpp
  template <typename T>
  T max(T a, T b) {
      return a > b ? a : b;
  }
  ```
* **Вызов:**
  * Явное указание: `max<int>(5, 10);`
  * Неявный вывод: `max(5, 10);` — компилятор сам выводит T = int.
* **Вывод аргументов (Template Argument Deduction):** Компилятор выводит параметры шаблона из типов аргументов. Работает только для функций (не для классов до C++17).
* **Ограничение вывода:** Если типы аргументов разные, вывод не сработает: `max(5, 10.5)` — ошибка (T не может быть одновременно int и double). Решения: явное указание `max<double>(5, 10.5)`, или несколько параметров шаблона, или `std::common_type`.
* **Несколько параметров:**
  ```cpp
  template <typename T, typename U>
  auto max(T a, U b) -> decltype(a > b ? a : b);  // C++11 trailing return type

  template <typename T, typename U>
  auto max(T a, U b) { return a > b ? a : b; }      // C++14 auto return
  ```

### Class Templates (Шаблоны классов)
* **Синтаксис:**
  ```cpp
  template <typename T>
  class Vector {
      T* data;
      size_t size;
  public:
      Vector();
      void push_back(const T& value);
      T& operator[](size_t i);
  };
  ```
* **Методы вне класса:** Требуют повторения `template <typename T>`:
  ```cpp
  template <typename T>
  Vector<T>::Vector() : data(nullptr), size(0) {}
  ```
* **Использование:** `Vector<int> v; Vector<std::string> vs;`
* **Тип класса:** `Vector` — это не тип, а "шаблон класса". `Vector<int>` — конкретный тип (инстанциация).
* **Вывод аргументов (CTAD, C++17):** Для классовых шаблонов компилятор может выводить типы из конструктора:
  ```cpp
  std::pair p(1, 2.5);  // C++17: выводится pair<int, double>
  ```
  Для этого нужны deduction guides (явные или неявные).

### Non-type Template Parameters (Нетиповые параметры)
* **Определение:** Параметры шаблона могут быть не только типами, но и значениями.
* **Синтаксис:** `template <typename T, size_t N> class Array { T data[N]; };`
* **Ограничения:** Целочисленные типы, перечисления, указатели, ссылки (до C++20). С C++20 — также литеральные типы (например, `double`, struct).
* **Пример:**
  ```cpp
  template <typename T, size_t N>
  constexpr size_t size(T (&)[N]) { return N; }
  // Выводит размер C-массива во время компиляции
  ```
* **Параметры по умолчанию:** `template <typename T = int, size_t N = 100>`

### Инстанцирование (Template Instantiation)
* **Неявное (Implicit):** Компилятор генерирует код при первом использовании с конкретным типом. Каждая единица трансляции может создать свою копию, линковщик объединяет.
* **Явное (Explicit):** `template class Vector<int>;` — принудительная генерация кода в данной единице трансляции. Ускоряет компиляцию.
* **extern template (C++11):** `extern template class Vector<int>;` — запрещает неявное инстанцирование в данной единице трансляции (будет использовать инстанцирование из другой единицы). Ускоряет компиляцию больших проектов.

### Специализация (Template Specialization)
* **Полная специализация (Explicit Specialization):** Определенная реализация для конкретного набора параметров.
  ```cpp
  template <>
  class Vector<bool> {
      // Битовое сжатие вместо массива
  };
  ```
  Для функций:
  ```cpp
  template <>
  const char* max(const char* a, const char* b) {
      return strcmp(a, b) > 0 ? a : b;
  }
  ```
* **Частичная специализация (Partial Specialization):** Специализация для подмножества параметров. Работает **только для классов** (не для функций).
  ```cpp
  template <typename T>
  class Vector<T*> {  // Частичная специализация для указателей
      //...
  };
  ```
* **Почему функции нельзя частично специализировать:** Перегрузка функций (overloading) покрывает те же случаи — компилятор может это сделать без отдельного механизма partial specialization.
* **Правила выбора:** Сначала ищется полная специализация, затем частичные, затем основной шаблон. Выбирается наиболее специфичный.

### SFINAE (Substitution Failure Is Not An Error)
* **Определение:** Если при подстановке типа в шаблон возникает некорректное выражение (например, обращение к несуществующему члену), это не ошибка компиляции. Компилятор просто исключает эту специализацию из рассмотрения и ищет другую.
* **Принцип:** "Ошибка подстановки — не ошибка" (Substitution Failure Is Not An Error). Ошибка подстановки — это когда происходит в сигнатуре функции (объявлении), а не в теле.
* **Применение:** Отключение перегрузок для неподходящих типов. Часто используется с `std::enable_if` и `decltype`.
* **Пример:**
  ```cpp
  template <typename T>
  auto foo(T t) -> decltype(t.some_method(), void()) {
      // работает только если у T есть some_method()
  }
  ```
* **std::enable_if (C++11):** Утилита для условного включения/отключения перегрузок.
  ```cpp
  template <typename T>
  std::enable_if_t<std::is_integral_v<T>, T>
  foo(T t) { return t * 2; }  // только для целочисленных T
  ```
* **Концепты (C++20):** Современная альтернатива SFINAE, делающая код читаемее:
  ```cpp
  template <std::integral T>
  T foo(T t) { return t * 2; }  // требует std::integral<T>
  ```

### Variadic Templates (Вариативные шаблоны, C++11)
* **Определение:** Шаблоны, принимающие произвольное количество аргументов.
* **Параметрический пакет (Parameter Pack):** `template <typename... Args>` — Args может быть нуль и более типов.
* **Распаковка (Pack Expansion):** `Args...` распаковывается в последовательность через запятую.
* **Пример:**
  ```cpp
  template <typename... Args>
  void print(Args... args) {
      (std::cout << ... << args);  // Fold expression (C++17)
  }
  ```
* **Рекурсивная распаковка (C++11):** До C++17 вариативные шаблоны обрабатывались рекурсивно:
  ```cpp
  void print() {}  // базовый случай (без аргументов)
  template <typename T, typename... Args>
  void print(T first, Args... rest) {
      std::cout << first;
      print(rest...);  // рекурсия с оставшимися аргументами
  }
  ```
* **sizeof...(Args):** Количество аргументов в пакете во время компиляции.

### Fold Expressions (C++17)
* **Определение:** Свертка параметрического пакета с бинарным оператором.
* **Формы:**
  * Унарная правая: `(args op ...)` → `(arg1 op (... op argN))`
  * Унарная левая: `(... op args)` → `((arg1 op arg2) op ...)`
  * Бинарная правая: `(args op ... op init)` → `(arg1 op (... op (argN op init)))`
  * Бинарная левая: `(init op ... op args)` → `((((init op arg1) op arg2) op ...) op argN)`
* **Примеры:**
  ```cpp
  template <typename... Args>
  auto sum(Args... args) { return (args + ...); }  // унарная правая
  template <typename... Args>
  void print(Args... args) { (std::cout << ... << args); }  // бинарная левая
  template <typename... Args>
  bool all(Args... args) { return (args && ...); }  // && свертка
  ```

### Template Template Parameters (Шаблонные параметры-шаблоны)
* **Определение:** Параметр шаблона, который сам является шаблоном.
* **Синтаксис:** `template <typename T, template<typename, typename> class Container>`
* **Пример:**
  ```cpp
  template <typename T, template<typename> class Container = std::vector>
  class Stack {
      Container<T> data;
  public:
      void push(const T& t) { data.push_back(t); }
  };
  // Использование: Stack<int, std::vector> s;  // но std::vector имеет 2 параметра (T, Allocator)
  ```
* **Проблема Allocator:** `std::vector<T, Allocator<T>>` — часто нужно подстраиваться под количество параметров контейнера.
* **Применение:** Адаптеры, обертки над контейнерами, политики (policy-based design).
* **typename vs class в template template:** Классический синтаксис: `template <typename T, template<typename> class Container>`. С C++17 можно: `template <typename T, template<typename> typename Container>`.

### Dependent Names (Зависимые имена)
* **Определение:** Если имя зависит от параметра шаблона, компилятор не может знать, что это (тип, функция, переменная) до инстанцирования.
* **Проблема:** Без подсказки компилятор считает зависимое имя не-типом (значением/функцией).
* **Решение: ключевое слово typename:** Нужно явно сказать, что зависимое имя — тип:
  ```cpp
  template <typename T>
  void foo() {
      typename T::iterator it;  // T::iterator — это тип
  }
  ```
* **template keyword:** Если зависимое имя само является шаблоном:
  ```cpp
  template <typename T>
  void foo() {
      T::template bar<int>();  // bar — это шаблонный метод
  }
  ```
* **Правило:** `typename` — перед зависимыми именами типов. `template` — перед зависимыми именами шаблонов (после `::`, `->`, `.`).

### Шаблонные методы в обычных классах
* **Определение:** Нешаблонный класс может иметь шаблонные методы-члены.
* **Синтаксис:**
  ```cpp
  class Printer {
  public:
      template <typename T>
      void print(const T& value) { std::cout << value; }
  };
  ```
* **Ограничение:** Шаблонные методы не могут быть виртуальными (vtable не может зависеть от шаблонного параметра, который инстанцируется в разных единицах трансляции).
* **Пример с умными указателями:** Шаблонный конструктор копирования для `shared_ptr`:
  ```cpp
  template <typename T>
  class shared_ptr {
  public:
      template <typename U>
      shared_ptr(const shared_ptr<U>& other);  // преобразование shared_ptr<Derived> -> shared_ptr<Base>
  };
  ```

### Краткие итоги по шаблонам
* **Генерация кода:** Шаблоны генерируют отдельный код для каждого набора параметров (если не использовать extern template или явное инстанцирование).
* **Размер бинарника:** Активное использование шаблонов может сильно увеличить размер исполняемого файла (code bloat).
* **Скорость компиляции:** Шаблоны замедляют компиляцию (каждая единица трансляции может инстанцировать одни и те же шаблоны).
* **Мощь:** Шаблоны позволяют писать эффективный обобщенный код без потери производительности (все инлайнится, нет накладных расходов на виртуальные вызовы).
* **Сложность:** Ошибки компиляции при работе с шаблонами часто очень длинные и трудночитаемые (особенно до C++20).
* **Концепты (C++20):** Значительно улучшают диагностику и читаемость, заменяя SFINAE в большинстве случаев.

## Лекция 9: Умные указатели и приведение типов

### 9.00 Умные указатели (Smart Pointers)

Умные указатели — классы, эмулирующие поведение сырых указателей, но с автоматическим управлением памятью. Основная цель — избежать утечек памяти и гарантировать корректное освобождение ресурсов (RAII).

#### auto_ptr (C++98, удалён в C++17)

* Первая попытка умного указателя в стандарте.
* **Проблема:** Копирование `auto_ptr` на самом деле перемещает владение (copy constructor принимает не-const ссылку и модифицирует источник). Нарушает семантику копирования.
* Иногда называли «бомбой замедленного действия» — код компилируется, но семантика неожиданная.
* Не работал с контейнерами STL (алгоритмы могли тихо терять данные).
```cpp
std::auto_ptr<int> p1(new int(42));
std::auto_ptr<int> p2 = p1;  // p1 теперь nullptr!
```

#### unique_ptr (C++11)

* **Эксклюзивное владение:** объект принадлежит ровно одному `unique_ptr`.
* **Только перемещение (move-only):** копирование запрещено (удалён copy constructor/assignment). Перемещение передаёт владение.
* **Zero-cost abstraction:** с умолчательным deleter'ом занимает ровно 1 указатель (как сырой указатель). Никаких накладных расходов.
* `std::make_unique<T>(args...)` (C++14) — предпочтительный способ создания. Исключает утечки при исключениях (по сравнению с `new T(args...)`).
* **API:** `.get()` — получить сырой указатель, `.reset(new_ptr)` — заменить объект (старый удаляется), `.release()` — открепить владение (возвращает сырой указатель, объект не удаляется).
* **Custom deleter:** можно передать функтор/lambda. Deleter — часть типа шаблона (`unique_ptr<T, Deleter>`), поэтому размер может увеличиться, если deleter с состоянием.
* **Специализация для массивов:** `unique_ptr<T[]>` — использует `delete[]`, есть operator[].
```cpp
std::unique_ptr<int> p = std::make_unique<int>(42);
auto raw = p.release();  // нужно вручную delete
p.reset(new int(10));    // старый объект удалён
```

#### shared_ptr (C++11)

* **Разделяемое владение:** объект жив, пока существует хотя бы один `shared_ptr`, указывающий на него.
* **Подсчёт ссылок:** внутри содержит счётчик ссылок (reference count), атомарный (thread-safe).
* `std::make_shared<T>(args...)` — предпочтительный способ. Делает **единое выделение памяти** под объект + control block (блок управления). Это эффективнее двух отдельных `new`.
* **Размер:** `shared_ptr` занимает 2 указателя (сырой указатель + указатель на control block).
* **Control block** содержит: reference count, weak count, deleter, allocator.
* **API:** `.use_count()` — текущее количество владельцев (не рекомендуется для принятия решений — может измениться), `.get()`, `.reset()`.
* **Атомарные операции:** копирование `shared_ptr` — потокобезопасно (атомарный инкремент счётчика).
```cpp
std::shared_ptr<int> p1 = std::make_shared<int>(42);
std::shared_ptr<int> p2 = p1;   // счётчик = 2
p1.reset();                      // счётчик = 1, объект жив
p2.reset();                      // счётчик = 0, объект удалён
```

#### weak_ptr (C++11)

* **Наблюдатель:** не владеет объектом, не увеличивает reference count (увеличивает weak count в control block).
* **Доступ к объекту:** `.lock()` — возвращает `shared_ptr` (если объект ещё жив) или `nullptr` (если объект уже удалён).
* **Проверка:** `.expired()` — проверяет, удалён ли объект.
* **Циклические ссылки:** решение проблемы взаимных `shared_ptr` (например, parent → child через `shared_ptr`, child → parent через `weak_ptr`).
```cpp
std::weak_ptr<int> w;
{
    auto s = std::make_shared<int>(42);
    w = s;
}  // объект удалён
auto s2 = w.lock();  // nullptr
```

#### enable_shared_from_this

* Класс-наследник от `std::enable_shared_from_this<T>` позволяет объекту получить `shared_ptr<T>` на самого себя.
* **Проблема:** если внутри метода сделать `shared_ptr<T>(this)`, создастся **второй независимый control block** для того же объекта → double-delete UB.
* **Решение:** `shared_from_this()` — возвращает `shared_ptr<T>`, использующий существующий control block. Работает только если объект уже управляется `shared_ptr` (иначе UB).
* **Принцип:** внутри `enable_shared_from_this` хранится `weak_ptr<T>`, который инициализируется при создании первого `shared_ptr`.

### 9.75 Приведение типов (Type Casting)

C++ предлагает 4 различных оператора приведения вместо единого C-style cast. Каждый — для своей цели.

#### static_cast<Type>(expr)

* **Компиляция:** проверка типов на этапе компиляции.
* **Безопасные преобразования:** int ↔ float, void* → T*, enum ↔ int.
* **Upcast:** приведение от наследника к базовому классу (безопасно, можно неявно, но static_cast тоже работает).
* **Downcast:** приведение от базового класса к наследнику — **без проверки на этапе выполнения**. Если реальный тип объекта не соответствует целевому — UB. В отличие от C-style cast, может изменить указатель при множественном наследовании (сдвиг адреса).
* **Не может** убрать const/volatile (для этого const_cast).
```cpp
float f = 3.14;
int x = static_cast<int>(f);            // 3
Base* b = new Derived();
Derived* d = static_cast<Derived*>(b);  // downcast, нет проверки
```

#### dynamic_cast<Type>(expr)

* **Runtime check:** проверяет реальный тип объекта на этапе выполнения.
* **Требование:** тип должен быть полиморфным (иметь виртуальную функцию / vtable). Использует RTTI.
* **Для указателей:** при неудаче возвращает `nullptr`.
* **Для ссылок:** при неудаче бросает `std::bad_cast`.
* **Cross-cast:** в иерархиях с множественным наследованием может приводить «вбок» (между разными базовыми классами одного объекта).
* **Стоимость:** медленнее static_cast (runtime-проверка).
```cpp
Base* b = new Derived();
Derived* d = dynamic_cast<Derived*>(b);
if (d) { /* ok */ }

Base& rb = *b;
Derived& rd = dynamic_cast<Derived&>(rb);  // может бросить bad_cast
```

#### const_cast<Type>(expr)

* **Назначение:** добавление или удаление cv-квалификаторов (const, volatile).
* **Опасность:** если исходный объект был объявлен const, попытка записи через снятый const — UB.
* **Типичный сценарий:** старый C-код с API, принимающим `char*`, но не изменяющим строку.
```cpp
const int x = 10;
int* p = const_cast<int*>(&x);
*p = 20;  // UB! x объявлен const
```

#### reinterpret_cast<Type>(expr)

* **Побитовая переинтерпретация:** трактует последовательность байт как другой тип.
* **Самый опасный:** нарушает strict aliasing rule (кроме случаев char*/byte*).
* **Implementation-defined:** результат может различаться на разных платформах.
* **Не может** удалить const/volatile (только reinterpret + const_cast).
```cpp
int x = 42;
float* p = reinterpret_cast<float*>(&x);  // опасно
```

#### C-style cast (Type)expr

* Пытается последовательно: `static_cast` → `const_cast` → `static_cast` + `const_cast` → `reinterpret_cast` → `reinterpret_cast` + `const_cast`.
* **Проблема:** непонятно, что именно делает (может делать слишком много).
* Рекомендуется **не использовать** в C++.

#### RTTI (Run-Time Type Information)

* `typeid(expr)` — возвращает `const std::type_info&`.
* `type_info::name()` — implementation-defined строка (часто искажённое имя типа).
* `type_info == type_info` — сравнение типов.
* `type_info::hash_code()` — хеш для использования в unordered контейнерах.
* Для работы нужна включенная RTTI (обычно включена по умолчанию, можно отключить флагом компилятора).
```cpp
Base& b = ...;
if (typeid(b) == typeid(Derived)) {
    // работает, но лучше dynamic_cast
}
```

#### Upcast vs Downcast

* **Upcast** (приведение к базовому классу): всегда безопасно, обычно неявно (или static_cast при необходимости, например для множественного наследования).
* **Downcast** (приведение к производному классу): требует static_cast (если уверены) или dynamic_cast (с проверкой).


## Лекция 10: Шаблоны (Templates) и обобщенное программирование
- **Идея Степанова**: Переход от микро-программирования к использованию стандартных компонентов с гарантированной сложностью (интерфейс + сложность).
- **Generic Programming**: Написание алгоритмов (например, `max`), которые работают для любых типов.
- **Требования шаблона**: Код шаблона неявно накладывает требования на тип `T`. Например, `max` требует наличия `operator>` и конструктора копирования. Шаблоны **не используют наследование**, это чистая кодогенерация.
- **Мономорфизация (Инстанцирование)**: Компилятор находит все используемые типы `T` (например, `int`, `Matrix`) и генерирует отдельную функцию под каждый.
  - *Плюсы*: "Zero-cost abstraction" (максимальная скорость), строгая проверка типов.
  - *Минусы*: Рост времени компиляции, раздувание бинарного кода (binary bloat), сложные ошибки.
- **Lazy Instantiation (Ленивость)**: Компилятор генерирует код методов класса только тогда, когда они вызываются. 
  - *Пример*: Если метод `Vector<T>::operator[]` содержит ошибку (обращается к полю, которого нет у `int`), класс `Vector<int>` скомпилируется успешно, пока вы не вызовете именно этот оператор.
- **Специализация**:
  - **Полная**: `template <>` — полная замена кода для конкретного типа.
  - **Порядок важен**: Специализация должна быть объявлена **до** первого инстанцирования, иначе возникнет ошибка.
  - **= delete**: Можно запретить использование шаблона для конкретных типов через специализацию.
- **Non-type template parameters**: Параметрами могут быть не только типы, но и константы (`int`, `size_t`). Пример: `FixedSizeVector<int, 16>`. Для каждого значения константы генерируется отдельный класс.
- **Compile-time execution**: Через шаблоны можно реализовывать рекурсию (числа Фибоначчи, возведение в степень). Компилятор кэширует результаты инстанцирования (мемоизация).
- **CRTP (Curiously Recurring Template Pattern)**:
  - Форма: `class Derived : public Base<Derived>`.
  - Используется для реализации **Mix-ins** (подмешивание поведения, например, счетчика объектов `Counter<T>`).
  - Позволяет реализовать **статический полиморфизм**: базовый класс вызывает методы наследника через `static_cast<Derived*>(this)`, что быстрее обычного `virtual`.

## Лекция 11: Реализация шаблонов, SFINAE и метапрограммирование
- **Two-phase name lookup (Двухфазный поиск)**:
  1. **Фаза 1 (до инстанцирования)**: Проверка имен, не зависящих от `T` (синтаксис, опечатки в глобальных функциях).
  2. **Фаза 2 (после инстанцирования)**: Проверка зависимых имен (зависят от `T`).
- **typename и template**:
  - Внутри шаблона `T::Bar` по умолчанию считается переменной. Чтобы сказать, что это тип, нужно писать `typename T::Bar`.
  - Если вызывается шаблонный метод зависимого объекта, нужно использовать `obj.template foo<T>()`.
- **SFINAE (Substitution Failure Is Not An Error)**:
  - Если при подстановке типа в шаблон функции получается невалидный код в сигнатуре (например, обращение к несуществующему `T::value_type`), компилятор не выдает ошибку, а просто исключает эту перегрузку из списка кандидатов.
- **Type Traits**: 
  - Реализуются через шаблоны и специализацию. `std::is_same<T, U>` наследуется от `false_type`, а его специализация `std::is_same<T, T>` — от `true_type`.
  - Позволяют проверять свойства типов (`is_integral`, `is_reference`) и модифицировать их (`remove_reference`).
- **std::conditional**: Выбор типа на этапе компиляции (аналог тернарного оператора для типов).
- **std::enable_if**: Инструмент для "выключения" функций. Если условие ложно, тип внутри `enable_if` не определяется, что вызывает SFINAE и исключает функцию.

## Лекция 12: Итераторы и вывод типов
- **Итераторы**: Абстракция над указателями. Позволяют обходить любые структуры данных (массивы, списки, деревья) единым способом.
  - Требуют перегрузки: `*`, `->`, `++`, `!=`, `==`.
  - Чтобы коллекция работала в `range-based for`, у нее должны быть методы `begin()` и `end()`.
- **Вывод типов (Deduction)**:
  - Компилятор выводит `T` автоматически по аргументам функции. 
  - **Важно**: При выводе не работают неявные приведения (`int` и `float` вместе не выведут `T`).
  - При выводе "по значению" ссылки (`&`) и константность (`const`) отбрасываются.
- **Value Categories (Категории значений)**:
  - **lvalue**: Имеет имя, можно взять адрес.
  - **prvalue**: Временный объект (результат функции, литерал).
  - **xvalue**: Объект, помеченный для перемещения (`std::move`).
  - **glvalue** = lvalue + xvalue; **rvalue** = prvalue + xvalue.
- **decltype(expr)**:
  - Если аргумент — имя переменной, возвращает ее тип как он есть.
  - Если аргумент — выражение, возвращает `T&` для lvalue, `T&&` for xvalue, `T` for prvalue.
- **decltype(auto)**: (C++14) Позволяет функциям возвращать ссылки или значения автоматически, сохраняя категорию значения.
- **CTAD**: (C++17) Вывод параметров шаблона класса из конструктора: `std::vector v = {1, 2}` вместо `std::vector<int>`.

## Лекция 13: Лямбды, Perfect Forwarding и Variadic Templates
- **Лямбда-выражения**: Синтаксический сахар для создания функциональных объектов.
  - Захват (`capture clause`): `[=]` (копия), `[&]` (ссылка), `[this]`.
  - Лямбды по умолчанию имеют `const operator()`. Чтобы менять захваченные переменные, нужно слово `mutable`.
  - **Захват с инициализацией**: `[p = std::move(ptr)]` (C++14).
  - Тип лямбды уникален и неизвестен, используется `auto` или `std::function`.
- **Универсальные ссылки (Forwarding References)**:
  - `T&&`, где `T` — выводимый тип. Может стать lvalue-ссылкой или rvalue-ссылкой.
  - **Reference Collapsing**: `&` + `&&` -> `&`. Единственный способ получить `&&` — это `&&` + `&&`.
- **Perfect Forwarding**: Использование `T&&` и `std::forward<T>(arg)`. Позволяет передать аргумент в другую функцию, не теряя его категорию (move остается move, copy остается copy).
- **Variadic Templates**: 
  - `template<typename... Args>` — пакет параметров.
  - `sizeof...(args)` — количество элементов в пакете.
  - **Fold Expressions (C++17)**: Компактная запись операций над пакетом. Бывают левые/правые и бинарные/унарные. Пример: `(args + ... + 0)`.
  - Раскрытие пакета: `args...` (раскрывает как `a1, a2, a3`).

## Лекция 14: constexpr и Концепты
- **constexpr**: 
  - Переменные: гарантированное вычисление при компиляции.
  - Функции: могут вычисляться при компиляции, если аргументы — константы. Если нет — работают как обычные функции (нет дублирования кода).
  - В C++20 разрешены циклы, `if constexpr`, локальные переменные и даже `new/delete` (транзитные аллокации).
- **consteval**: (C++20) Функция ВСЕГДА должна вычисляться при компиляции.
- **if constexpr**: (C++17) Ветвление при компиляции. Невыбранная ветка не компилируется (игнорируется). Позволяет заменять сложные SFINAE-конструкции.
- **Пользовательские литералы**: `10_x + 14_y` — создание объектов через суффиксы.
- **Concepts (C++20)**:
  - Формализованные требования к типам. 
  - `requires` clause: накладывает ограничение. `requires` expression: проверяет компилируемость кода.
  - Позволяют перегружать функции по концептам (например, одна версия для целых чисел, другая для дробных).
  - Радикально улучшают сообщения об ошибках.

## Лекция 15: Инициализация и умные указатели
- **Инициализация**:
  - Существует много видов (Default, Value, Direct, Copy, Aggregate, List).
  - **Uniform Initialization**: Фигурные скобки `{}` предотвращают сужение типов (narrowing) и решают проблему "most vexing parse".
  - **std::initializer_list**: Если у класса есть конструктор от этого типа, фигурные скобки `{}` всегда выберут его.
- **Умные указатели (Smart Pointers)**:
  - **std::unique_ptr**: Монопольное владение. Не копируется. Можно передать кастомный `Deleter` (для закрытия файлов, сокетов и т.д.). Размер равен обычному указателю (если делетер — пустой класс).
  - **std::shared_ptr**: Разделяемое владение.
    - Использует **Control Block**, где лежат два счетчика (сильный и слабый) и указатель на объект (или сам объект).
    - `make_shared` аллоцирует блок управления и объект одним куском памяти — это быстрее и экономнее.
  - **std::weak_ptr**: Решает проблему **циклических ссылок** (когда объекты держат друг друга и не удаляются). Не владеет объектом, проверяет его наличие через `.lock()`.
  - **enable_shared_from_this**: Позволяет объекту, уже владеемому через `shared_ptr`, безопасно выдать новый `shared_ptr` на самого себя (через `weak_ptr` внутри).
